# Step 6. 통합 및 레거시 정리 (Plan)

Written at (KST): 2026-07-14 10:04

## 1. 작업 목적

`catalog`/`rules`/`builder`/`wizard`/`cli` 5개 모듈이 모두 완성되어, 새 구조가 기존
`assembly.py`와 동일한 기능을 제공할 준비가 됐다. 이제 `assembly.py`를 새 패키지를
호출하는 얇은 진입점으로 축소하고, 기존 절차지향 코드(전역 변수, 중복 if-elif)를
제거해 리팩토링을 마무리한다. 또한 사용자가 요청한 대로, 사용법과 고수준 동작 흐름을
담은 `README.md`를 작성한다.

## 2. 현재 상태 (리더 관점)

- `assembly.py`에는 여전히 리팩토링 이전의 260줄짜리 절차지향 코드가 그대로 남아 있다.
- 새로 만든 `car_assembly` 패키지는 완성되어 테스트도 통과하지만, 실제로 실행되는
  진입점(`assembly.py`)과는 아직 연결되어 있지 않다.
- 이 프로젝트를 처음 보는 사람이 참고할 `README.md`가 없어, 어떻게 실행하는지 알기 어렵다.

## 3. 목표 상태

- `assembly.py`는 `car_assembly.cli.main()`을 호출하는 몇 줄짜리 진입점으로 바뀐다.
- 기존 전역 변수/중복 분기 코드는 저장소에서 사라진다.
- `README.md`를 열면 (1) 이 프로그램이 무엇인지, (2) 어떻게 실행하는지, (3) 사용자가
  어떤 순서로 조작하게 되는지(고수준 흐름)를 몇 분 안에 파악할 수 있다.

## 4. 대표 시나리오

- 새로 저장소를 받은 사람이 `README.md`만 읽고 `python assembly.py`를 실행해 차량을
  조립하고 Run/Test까지 해볼 수 있다.
- 실제로 터미널에서 실행해보면, 리팩토링 전과 화면 문구/동작이 동일하게 보인다.

## 5. 변경 접근 방식

- `assembly.py`를 `from car_assembly.cli import main` + `if __name__ == "__main__": main()`
  형태로 교체한다.
- `assembly.py`가 `car_assembly.cli.main`을 그대로 가리키는지 확인하는 최소 테스트를 추가한다.
- `README.md`에는 코드 수준 설명 대신, 사용자가 겪는 흐름(타입 선택 → 부품 선택 → 완성 →
  Run/Test) 위주의 고수준 흐름도(예: 간단한 화살표 다이어그램)와 실행 방법을 담는다.

## 6. 예상 설명 구조

- README: 프로젝트 한 줄 소개 → 실행 방법(가상환경/설치/실행 명령) → 고수준 동작 흐름도 →
  대표 예시(정상 조합 Run 성공, 비호환 조합 Run 실패) 순으로 구성한다.
- 코드 파일 목록이나 함수별 설명은 README에 넣지 않는다(그 내용은 각 Step의 plan/result
  문서에 이미 있다).

## 7. 가정 및 리스크

- 가정: README의 "고수준 흐름도"는 함수 호출 순서가 아니라 사용자가 화면에서 겪는 단계
  (차량 타입 → 엔진 → 제동장치 → 조향장치 → 완성 → Run/Test)를 기준으로 그린다.
- 리스크: 낮음. 이미 검증된 모듈을 연결하고 죽은 코드를 제거하는 작업이라 새 버그가 생길
  가능성은 낮지만, 마지막에 실제 터미널 실행으로 눈으로 직접 확인한다.

## 8. 검증 방법

- `python -m pytest -q`로 전체 테스트(기존 43개 + 신규 진입점 테스트) 통과를 확인한다.
- 터미널에서 `python assembly.py`를 직접 실행(또는 입력을 스크립트로 주입)해, 차량 타입
  선택부터 Run/Test까지 한 번씩 수동으로 확인한다.
