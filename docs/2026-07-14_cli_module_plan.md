# Step 5. cli.py — I/O 계층 (Plan)

Written at (KST): 2026-07-14 10:04

## 1. 작업 목적

지금까지 만든 `catalog`/`rules`/`builder`/`wizard`는 모두 화면 출력이나 사용자 입력을
다루지 않는 순수 로직이다. 이 로직들을 실제로 사용해 메뉴를 보여주고, 입력을 받고,
Run/Test를 실행하는 화면(터미널) 계층을 만든다. `input()`/`print()`가 등장하는 곳은
이 모듈 하나뿐이도록 만드는 것이 핵심이다.

## 2. 현재 상태 (리더 관점)

- 기존 `assembly.py`는 화면 출력, 입력 처리, 상태 변경, 규칙 검사가 `main()` 안에 모두
  뒤섞여 있어 어디까지가 "화면"이고 어디까지가 "로직"인지 구분하기 어렵다.
- 그 결과 로직만 따로 테스트할 방법이 없었다(이전 4단계로 그 문제는 이미 해결됨).
  이번 단계는 그 로직들을 화면과 연결하는 "마지막 조립" 단계다.

## 3. 목표 상태

- `car_assembly/cli.py` 하나에만 `input()`/`print()`가 존재한다.
- 메뉴에 표시되는 번호 목록(예: "1. GM", "2. TOYOTA"...)은 `catalog.py`의 옵션 데이터를
  그대로 순회해서 만들어지므로, 새 부품을 추가하면 메뉴에도 자동으로 나타난다.
- 기존 화면 문구(안내 문구, 오류 문구, 선택 확인 문구, Run/Test 결과 문구)는 한 글자도
  바꾸지 않는다 — 사용자가 보는 화면은 리팩토링 전후로 동일해야 한다.

## 4. 대표 시나리오

- 사용자가 "차량 타입 → 엔진 → 제동장치 → 조향장치"를 순서대로 선택하고 Run을 실행하면,
  기존과 동일한 형식의 "Car Type : Truck" / "Engine   : GM" ... 요약과 "자동차가
  동작됩니다." 문구가 출력된다.
- 호환되지 않는 조합으로 Run을 실행하면 "자동차가 동작되지 않습니다"만 출력되고 멈춘다.
- 여러 규칙을 동시에 어긴 조합으로 Test를 실행하면, 기존과 동일하게 **첫 번째로 위반된
  규칙 하나만** "FAIL\n<사유>" 형식으로 출력된다.
- 숫자가 아닌 값을 입력하면 "ERROR :: 숫자만 입력 가능"이 출력된다.
- 완성 단계에서 고장난 엔진을 선택한 채 Run을 실행하면, "엔진이 고장나있습니다." /
  "자동차가 움직이지 않습니다."가 출력된다(이 조합 자체는 호환성 규칙 위반이 아니므로
  별도로 확인해야 한다는 기존 동작을 그대로 유지).

## 5. 변경 접근 방식

- 메뉴 출력(`show_menu`)은 단계별 고정 문구(제목, ASCII 아트, "0. 뒤로가기" 등)만 직접
  출력하고, 번호가 매겨진 선택지 목록은 `wizard.options()`를 순회해 만든다.
- 선택 확인 문구, 단계별 오류 문구는 조회 테이블(딕셔너리)로 관리해 기존 문구를 그대로
  재현한다. (예: 고장난 엔진을 선택했을 때 "고장난 엔진 엔진을 선택하셨습니다"처럼 단어가
  겹치지 않도록, 확인 문구는 부품별로 개별 매핑한다.)
- `run_produced_car(config)` / `test_produced_car(config)`는 `rules.py`의
  `is_valid`/`find_violations`만 사용해 판정하고, 화면에 보여줄 요약 라벨은 기존 문구의
  대소문자 표기(예: 제동장치/조향장치 요약에서만 "Mando"/"Bosch"처럼 첫 글자만 대문자로
  쓰던 기존 표기)를 그대로 보존한다.
- `main()`이 `Wizard`를 만들어 입력 루프를 돌리며, 뒤로가기/단계 진행/Run/Test 분기를
  `wizard`의 메서드로 처리한다.

## 6. 예상 설명 구조

- 별도 다이어그램 없이 대표 시나리오로 충분히 설명 가능하다.

## 7. 가정 및 리스크

- 가정: 화면에 보이는 모든 문구(안내/오류/확인/결과)는 기존 `assembly.py`와 완전히 동일하게
  유지한다. 이는 "동작 보존"이 리팩토링의 전제이기 때문이며, 기존 코드에 있던 표기 불일치
  (예: 제동장치 요약에서만 소문자 시작 표기를 쓰는 것)도 의도적으로 그대로 둔다 — 이를
  "통일"하는 것은 이번 리팩토링 범위를 벗어나는 별도의 변경이기 때문이다.
- 리스크: 중간. `input()`/`print()`를 다루는 계층이라 완전한 단위 테스트는 어렵고, 표준
  입출력을 흉내 낸(monkeypatch) 통합 테스트 형태로 검증한다.

## 8. 검증 방법

- TDD로 진행: 오류/확인 문구 조회, Run/Test 결과 출력(정상/위반/고장난 엔진 케이스),
  `main()`의 전체 흐름(입력 시나리오를 스크립트로 주입)을 테스트로 먼저 작성해 실패를
  확인한 뒤 구현한다.
- `python -m pytest -q`로 전체 테스트 통과를 확인하고, 마지막으로 실제 터미널에서
  수동으로 한 번 실행해 화면이 기존과 동일하게 보이는지 눈으로 확인한다.
