# Step 1. catalog.py — 차량/부품 카탈로그 데이터 (Plan)

Written at (KST): 2026-07-14 10:04

## 1. 작업 목적

기존 `assembly.py`는 "1이면 GM, 2면 TOYOTA..." 같은 라벨 매핑과 유효 범위를 여러 함수
(`show_menu`, `is_valid_range`, `select_engine`, `run_produced_car` 등)에서 각자 if-elif로
반복 구현하고 있다. 차량 타입/부품 제조사라는 카탈로그 데이터를 코드 한 곳으로 모아,
이후 단계(rules/builder/wizard/cli)가 이 데이터를 참조하도록 만든다.

## 2. 현재 상태 (리더 관점)

- 차량 타입, 엔진/제동장치/조향장치 제조사 목록이 여러 함수에 흩어져 있어, 새 항목을
  추가하려면 어느 함수를 고쳐야 하는지 한눈에 알기 어렵다.
- 번호(1/2/3...)와 이름(GM, TOYOTA...)의 대응 관계가 코드 곳곳에 문자열로 하드코딩되어 있다.

## 3. 목표 상태

- `car_assembly/catalog.py` 한 곳에 차량 타입/부품 제조사 데이터가 선언된다.
- 각 항목은 "사용자가 입력하는 번호(answer) - 내부 값(value) - 화면 라벨(label)"을 한 세트로 갖는다.
- 이후 다른 모듈은 이 데이터를 조회만 하면 되고, 새 항목 추가는 이 파일에 한 줄 추가하는 것으로 끝난다.

## 4. 대표 시나리오

- `find_option(CAR_TYPE_OPTIONS, 2)` 호출 시 SUV에 해당하는 옵션(번호 2, 라벨 "SUV")을 돌려준다.
- 존재하지 않는 번호(예: 99)를 조회하면 `None`을 돌려준다 (호출 측에서 안전하게 분기 가능).
- `CarConfig()`를 새로 만들면 타입/엔진/제동장치/조향장치가 모두 선택되지 않은 초기 상태다.

## 5. 변경 접근 방식

- `CarType`, `EngineMaker`, `BrakeMaker`, `SteeringMaker` 4개의 `Enum`으로 부품/타입의 정체성을 표현한다.
- 각 카테고리마다 `Option(answer, value, label)` 목록을 원래 메뉴 순서 그대로 선언한다
  (엔진의 경우 스펙에 있던 "4. 고장난 엔진"도 하나의 선택지로 포함한다).
- `find_option(options, answer)` 헬퍼로 번호 → 옵션 조회를 통일한다.
- 부품 선택 상태를 담는 `CarConfig` 데이터 클래스를 함께 정의한다 (다음 단계인 `rules.py`가
  위반 여부를 판단할 때 이 타입을 입력으로 받는다).

## 6. 예상 설명 구조

- 표/다이어그램 없이, 데이터 선언과 조회 함수만으로 충분히 이해 가능한 모듈이라 별도 구조는 두지 않는다.

## 7. 가정 및 리스크

- 가정: "고장난 엔진"은 별도 예외 케이스가 아니라 `EngineMaker`의 정상적인 한 옵션으로 취급한다
  (기존 코드에서도 4번으로 선택 가능한 메뉴 항목이었다).
- 가정: `CarConfig`는 아직 선택되지 않은 필드를 `None`으로 표현하는 가변(mutable) 데이터 클래스로 만든다
  (다음 단계의 `CarBuilder`가 단계별로 값을 채워 넣어야 하기 때문).
- 리스크: 낮음. 순수 데이터 정의라 기존 동작에 영향이 없다.
