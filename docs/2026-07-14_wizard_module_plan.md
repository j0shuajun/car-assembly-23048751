# Step 4. wizard.py — 단계 전이 로직 (Plan)

Written at (KST): 2026-07-14 10:04

## 1. 작업 목적

기존 `assembly.py`는 `step`이라는 정수 하나로 진행 단계를 표현하고, 이 값에 대한 if-elif
분기가 `show_menu`, `is_valid_range`, `main`의 while 루프 세 곳에 중복되어 있다. 단계
정의와 전이(다음 단계로 진행 / 이전 단계로 뒤로가기) 규칙을 `Wizard` 하나로 모아,
새 단계가 추가되어도 한 곳만 고치면 되게 만든다.

## 2. 현재 상태 (리더 관점)

- "차량 타입 선택(0) → 엔진(1) → 제동장치(2) → 조향장치(3) → 완성(4)"이라는 같은 순서
  정보가 세 함수에 중복 표현되어 있다.
- "몇 번 단계에서 0을 입력하면 뒤로가기가 되는지", "완성 단계에서 0을 입력하면 왜 처음
  화면(0단계)으로 바로 돌아가는지"같은 규칙이 `main()`의 while 루프 안에 흩어져 있어 파악이
  어렵다.

## 3. 목표 상태

- `Wizard`가 현재 단계, 현재 단계에 필요한 카탈로그 옵션, 유효한 답변 범위, 다음 단계로
  진행/이전 단계로 되돌아가는 규칙을 모두 캡슐화한다.
- 새 단계를 추가하려면 단계 순서 목록과 옵션 매핑에 항목을 추가하는 것만으로 끝난다.

## 4. 대표 시나리오

- 새로 만든 `Wizard`는 "차량 타입 선택" 단계에서 시작하며, 이 단계에서는 0(뒤로가기)을
  입력할 수 없다 (원본 스펙과 동일하게 첫 단계에는 뒤로가기가 없다).
- 엔진 선택 단계에서 유효한 번호로 `select()`를 호출하면 상태가 갱신되고 다음 단계(제동장치)로
  자동으로 넘어간다.
- 엔진/제동장치/조향장치 단계에서 0을 입력하면 바로 이전 단계로 한 칸 되돌아간다.
- "완성" 단계에서 0을 입력하면 (이전 단계인 조향장치가 아니라) 처음 화면인 차량 타입
  선택 단계로 돌아간다 — 기존 `main()`의 `if step == 4: step = 0` 규칙과 동일하다.

## 5. 변경 접근 방식

- `Step` Enum(`CAR_TYPE`, `ENGINE`, `BRAKE`, `STEERING`, `FINISH`)으로 단계를 표현한다.
- 단계별 카탈로그 옵션과 `CarBuilder`의 대응 선택 메서드를 매핑 테이블로 연결한다.
- `Wizard.is_valid_answer(answer)`가 현재 단계 기준 유효 범위를 판단하고,
  `Wizard.select(answer)`가 선택을 적용하며 다음 단계로 이동하고,
  `Wizard.go_back()`이 이전 단계로 되돌아간다 (완성 단계는 예외적으로 첫 단계로 이동).
- "완성" 단계(Run/Test 선택)는 카탈로그 옵션이 없으므로 `options()`가 `None`을 돌려주고,
  실제 Run/Test 실행은 이 모듈의 책임이 아니다(Step 5 `cli.py`에서 처리).

## 6. 예상 설명 구조

- 별도 다이어그램 없이 대표 시나리오와 표로 충분히 설명 가능하다.

## 7. 가정 및 리스크

- 가정: "완성" 단계에서 0을 입력했을 때 처음 화면(차량 타입 선택)으로 돌아가는 기존 동작을
  그대로 유지한다(조향장치 단계로 돌아가지 않음 — 기존 `main()` 로직과 동일).
- 리스크: 낮음. 상태 전이 규칙만 옮기는 작업이며, 화면 출력/Run·Test 실행 로직은 포함하지 않는다.
